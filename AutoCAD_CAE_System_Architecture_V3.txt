AutoCAD/CAE 系统架构 V3（确认版）

本文档基于当前项目目标修订，替代 V2 的关键歧义点。目标是建立一条“检索优先、LLM兜底、分阶段审查、可持续沉淀”的自动化工程计算流水线。

--------------------------------
一、V3 核心结论（已确认）
--------------------------------

1) 输入路由：先检索，后生成
- 客户输入可为：自然语言、图片、STEP 文件等。
- 系统先在 Template Library / Project Case Library 筛选匹配。
- 命中则优先复用（参数化套用）；未命中再由 LLM 生成。

2) LLM 生成+自修复为主线机制
- 各关键阶段允许 LLM 直接生成脚本，并在受控边界内自动修复。
- 不是可选附加功能，而是主执行能力之一。

3) 不使用 FEniCSx
- 不引入 FEniCSx 求解链路、依赖或 fallback。
- 求解主线保持 CalculiX 路径（或既定非 FEniCSx 路线）。

4) 保留交互审查
- CAD、Gmsh 等关键阶段必须经过：
  自检 -> 可视化 -> 客户确认（confirm/edit/abort）-> 下一阶段
- 默认不允许“无确认直通全流程”。

--------------------------------
二、系统总体目标
--------------------------------

构建可工程化落地的自动化 CAE 流水线：

客户需求（NL/图片/STEP）
-> 检索复用（Template/Project Case）
-> 必要时 LLM 生成脚本
-> 分阶段自检与人工确认
-> 求解与后处理
-> 案例沉淀与模板孵化

系统最终支持：
- 客户项目自动分析
- 批量运行与可追溯审计
- 模板持续演化
- 真实工程案例复现

--------------------------------
三、V3 主流程
--------------------------------

Stage A: Intake（多模态接入）
- 接收需求文本、图像、STEP、约束条件等。

Stage B: Retrieval（检索优先）
- 在 Template Library / Project Case Library 进行匹配和排序。
- 输出：推荐路径（复用或新建）。若复用，则包括：当前案例下cad文件模板、gmsh网格划分规则建议、solver默认边界条件及求解设置。

Stage C: CaseSpec 统一化
- 无论来源如何，统一转为标准 CaseSpec 进入流水线。 CaseSpec的输出格式为ｊｓｏｎ，包括当前案例下cad文件（.step)、gmsh网格划分规则建议、solver默认边界条件及求解设置建议等。

Stage D: CAD（LLM 脚本主线）
- 生成/修复 CAD 脚本 -> 执行 -> 几何自检 -> 几何预览 -> 客户确认。输出.step格式文件。

Stage E: Mesh/Gmsh（LLM 脚本主线）
- 生成/修复网格脚本 -> 执行 -> 网格质量自检 -> 网格预览 -> 客户确认。

Stage F: Solver（非 FEniCSx）
- 生成求解输入 -> 执行求解 -> 收敛/错误诊断。

Stage G: Output/Postprocess
- 输出结果摘要、图表、关键指标、诊断与产物清单。

Stage H: 沉淀闭环
- 合格 run 进入 Project Case Library。
- 达到条件触发 Template Candidate 审核与提升。

--------------------------------
四、交互审查（Review Gate）规则
--------------------------------

关键阶段统一状态机：
auto_check -> preview -> user_confirm(confirm/edit/abort) -> next

CAD Gate：
- 几何完整性、拓扑有效性、尺寸合理性检查。

Mesh Gate：
- 单元质量、物理组完整性、边界命名映射检查。

Solver Gate（可配置）：
- 输入文件完整性、求解可执行性、风险提示检查。

--------------------------------
五、LLM 自修复控制边界
--------------------------------

- 必须有界重试：max_attempts。
- 必须有限错误域：failure class filter。
- 必须可追溯：记录每轮脚本、日志、错误归因与修复动作。
- 禁止无边界全链路自动修复。

--------------------------------
六、库与治理模型
--------------------------------

保留四库架构：
1) Template Library
2) Project Case Library
3) Material Library
4) Feature Library

关键治理字段保留：
- template_affinity（full/partial/none）
- template_link（关联模板ID）
- template version（版本可复现）

--------------------------------
七、数据与产物最小规范
--------------------------------

- 标准输入：CaseSpec
- 运行索引：runs/index.jsonl（append-only）
- 失败归因：issue_report.json
- 审查记录：review_transcript.json
- 预览产物：CAD PNG、Mesh PNG
- 结果产物：summary/diagnostics/plots/csv/json

--------------------------------
八、本版与 V2 的关键差异
--------------------------------

1) 明确“检索优先、LLM兜底”的输入策略。
2) 明确“LLM 生成+自修复”为主线机制。
3) 明确排除 FEniCSx。
4) 明确分阶段交互审查为默认必经流程。
