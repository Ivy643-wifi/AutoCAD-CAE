## Project Overview

**AutoCAE Pipeline** is an automated CAD/CAE analysis system (Phase 1 MVP) that orchestrates an 8-stage engineering analysis workflow:

```
CaseSpec в†’ CadQuery в†’ Gmsh в†’ AnalysisModel в†’ CalculiX в†’ Postprocess в†’ Output
```

**Core principles (from PROJECT_RULES.md V3):**
- **Retrieval-first**: Template Library and Project Case Library search before LLM generation
- **Controlled LLM generation**: Scripts generated by LLM with bounded retry self-repair
- **Interactive review gates**: CAD and Mesh stages require user confirmation before proceeding
- **Full traceability**: All decisions and artifacts recorded for audit and replay
- **No FEniCSx**: Strictly prohibited; CalculiX is the solver

## Common Commands

### Running the Pipeline

```bash
# Full 8-stage pipeline (with interactive review gates)
autocae run examples/flat_plate_tension.yaml

# Dry-run (skip actual CalculiX solving, stages 0-5 only)
autocae run examples/flat_plate_tension.yaml --dry-run

# With LLM-based CAD generation instead of templates
autocae run examples/flat_plate_tension.yaml --cad-mode llm --cad-llm-max-attempts 3

# With external STEP file (skip CadQuery, dual-track mode)
autocae run examples/flat_plate_tension.yaml --step-file /path/to/model.step
```

### Intake & Review Commands

```bash
# Intake (retrieval-first routing): text input в†’ CaseSpec + intake_decision.json
autocae intake --text "flat plate tension length=200 width=25 thickness=2"

# Intake from STEP file
autocae intake --step-file /path/to/model.step

# Interactive review gates (CAD + Mesh validation before solving)
autocae review runs/<case_id>/ --stage all

# CAD gate only
autocae preview cad runs/<case_id>/ --decision confirm

# Mesh gate only
autocae preview mesh runs/<case_id>/ --decision confirm
```

## Architecture Overview

### 8-Stage Pipeline (orchestrator/pipeline.py)

```
Stage 0: Validate       в†’ CaseSpecValidator (business rules)
Stage 1: TemplateMatch  в†’ TemplateRegistry (retrieval-first)
Stage 2: CAD            в†’ CADService or LLM generation + auto-repair
Stage 3: CAD Gate       в†’ CadGateService (auto_check в†’ preview в†’ user_confirm)
Stage 4: Mesh           в†’ MeshService or LLM generation + auto-repair
Stage 5: Mesh Gate      в†’ MeshGateService (auto_check в†’ preview в†’ user_confirm)
Stage 6: SolverInput    в†’ CalculiXAdapter (AnalysisModel в†’ solver deck)
Stage 7: SolverRun      в†’ SolverRunner (CalculiX execution or dry_run)
Stage 8: Postprocess    в†’ PostprocessEngine (result extraction + diagnostics)
```

All inter-stage data flows through **file interfaces** (JSON, STEP, mesh.inp, job.inp) living in `runs/<case_id>/`.

### Key Modules

| Module | Purpose |
|--------|---------|
| `schemas/` | Pydantic models: CaseSpec, AnalysisModel, MeshGroups, RunStatus, ResultSummary |
| `backend/input/` | Input loading and validation (YAML/JSON в†’ CaseSpec) |
| `backend/templates/` | Template registry, matching, instantiation (retrieval-first path) |
| `backend/services/` | Stage executors: CAD, Mesh, Solver, LLM-based generation/repair, visualization |
| `backend/review/` | Review gates (cad_gate, mesh_gate) with checks and transcripts |
| `backend/orchestrator/` | PipelineRunner (8-stage orchestration), ArtifactLocator (run directory compatibility) |
| `backend/intake/` | Intake service (retrieval + routing logic) |
| `cli.py` | Typer CLI entry point (user-facing commands) |

### Core Patterns

**1. Retrieval-First Intake (M1.1)**
- Input (text/STEP/image) в†’ IntakeService
- Search Template Library and Project Case Library
- If confidence в‰Ґ threshold в†’ parameterized reuse (output: CaseSpec)
- Else в†’ LLM generation (forward to CAD stage)
- Always output: `intake_decision.json` (with route, candidates, confidence)

**2. LLM Script Generation + Bounded Auto-Repair**
- CAD stage: LLM generates CadQuery script в†’ execute в†’ if error, LLM repairs (max 3 attempts)
- Mesh stage: Same pattern with Gmsh scripts
- Config: `max_attempts`, `failure_class_filter`, `stop_conditions`
- Audit trail: Every iteration logged (script, error, repair, result)

**3. Interactive Review Gates**
- After CAD: auto_check в†’ preview PNG в†’ user decision (confirm/edit/abort)
- After Mesh: Same, blocks solver until confirmed
- Records: `review_transcript.json` (decision, comment, edit_request, timestamp)
- If abort в†’ pipeline stops; if edit в†’ human modifies and reruns stage

**4. Run Directory Contract (M1.9)**
- **Canonical (preferred):** `runs/<case_id>/file.ext` (flat)
- **Legacy (read-only):** `runs/<case_id>/02_cad/`, `03_mesh/`, etc. (subdir)
- **Accessor:** Always use `ArtifactLocator` for compatibility
- Standard files: case_spec.json, model.step, mesh.inp, solver_job.json, run_status.json, review_transcript.json, issue_report.json, field_manifest.json, diagnostics.json

**5. Audit & Traceability**
- Every run produces `runs/index.jsonl` (append-only timeline)
- Each run: `issue_report.json` (error_stage, root_cause_hint, remediation_hint)
- Gates: `review_transcript.json` (all user decisions timestamped)
- LLM repairs: Full iteration history in solver/analysis logs

## Important Constraints & Rules

**From PROJECT_RULES.md (V3):**

1. **Never add FEniCSx** вЂ” Prohibited by hardcoded rule; CalculiX only
2. **Main pipeline fixed:** CaseSpec в†’ CadQuery в†’ Gmsh в†’ CalculiX в†’ Output (no reordering without Master Control decision)
3. **Templates as first resort** вЂ” Retrieval must precede generation; templates are parameterized before LLM invocation
4. **Bounded retries** вЂ” All LLM loops must have max_attempts, failure filters, and stop conditions
5. **User gate confirmations** вЂ” CAD and Mesh stages are NOT auto-pass; they require explicit user decision
6. **Interface stability** вЂ” Core JSON objects (case_spec.json, analysis_model.json, solver_job.json, etc.) are "frozen"; changes require impact statement
7. **Modular boundaries** вЂ” Each module (CaseSpec, CadQuery, Gmsh, CalculiX, Output) has strict input/output contracts; no cross-pollution

## Notes for Future Work

1. **M1 completion checklist:** Intake (routing logic), CAD Gate, Mesh Gate, CAD LLM repair, Mesh LLM repair, review command, audit artifacts (issue_report.json, index.jsonl), doctor, artifact_locator
2. **Interface stability:** Before modifying case_spec.json, analysis_model.json, solver_job.json, etc., file an impact statement (see PROJECT_RULES.md В§ 12)
3. **LLM service patterns:** Both cad_llm_service.py and mesh_llm_service.py follow identical repair strategies; if one changes, check the other
4. **Review gates:** Not optional; both CAD and Mesh gates must be passed before proceeding to next stage; bypass only with explicit --no-enforce-* flag and documented reason
5. **Templates vs LLM:** Template path (stage 2 or 4) is always preferred; LLM generation only when template search fails or user selects --cad-mode llm / --mesh-mode llm
